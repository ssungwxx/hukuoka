# Hukuoka Schedule Manager

후쿠오카 4박 5일 여행 일정(2026년 3월 30일 ~ 4월 3일)을 관리한다.

## Data Location

모든 일정 데이터는 `/Users/felix/personal/projects/hukuoka/src/app/page.tsx`에 TypeScript 상수로 하드코딩되어 있다. 이 파일을 항상 먼저 읽어서 현재 상태를 파악한 뒤 작업할 것.

### 핵심 데이터 구조

**일별 아이템 배열** (`day1Items` ~ `day5Items`):
```typescript
type ScheduleItem = {
  time: string;       // "HH:MM" 형식
  emoji: string;      // 이모지 1개
  text: string;       // 일정 제목
  detail?: string;    // 상세 설명
  cost?: string;      // 비용 (예: "¥1,500~2,500")
  location?: { lat: number; lng: number };  // 지도 좌표
  restaurant?: {
    tabelog?: { rating: string; url: string };
    google?: { rating: string; url: string };
    signature?: string;  // 대표 메뉴
  };
};
```

**DaySchedule 컴포넌트 props**: `day`, `date`, `title`, `transportCost`, `mapUrl`, `items`, `note`, `defaultOpen`

**교통 정보** (`transportRoutes` 배열): `from`, `to`, `method`, `time`, `cost`

**식사 테마 배열** (`mealData`): `day`, `meal`, `theme`, `restaurant`

## Hard Constraints

일정을 수정할 때 아래 제약 조건을 반드시 검증해야 한다. 위반 시 사용자에게 경고하고 대안을 제시할 것.

## Workflow

### 1. 일정 조회 (View)

사용자가 "Day N 일정 보여줘", "타임라인" 등을 요청하면:

1. `page.tsx`를 읽어 해당 날의 `dayNItems` 배열을 파싱한다
2. 아래 형식으로 깔끔한 타임라인을 출력한다:

```
## Day 1 - 3/30 (Mon) : 도착 & 야타이 체험

| 시간 | 일정 | 비고 |
|------|------|------|
| 17:00 | ✈️ 후쿠오카 공항 도착 | 입국심사 30~45분 |
| 17:45 | 🚌 국제선→국내선 셔틀 | 15분 소요 |
| 18:00 | 🚇 지하철 → 하카타역 | ¥260, 5분 |
| ... | ... | ... |

**교통비 합계**: ¥260
**잠재 이슈**: (있으면 표시)
```

3. 시간 간격이 15분 미만인 연속 일정이 있으면 "빡빡한 전환" 경고를 표시한다
4. 이동시간이 고려되지 않은 구간이 있으면 지적한다

### 2. 일정 추가 (Add)

1. 사용자가 장소/시간/활동을 알려주면, `page.tsx`를 읽어 현재 일정을 확인한다
2. 시간 충돌 검사:
   - 기존 아이템과 시간이 겹치는지 확인
   - 이전 아이템에서의 이동시간을 `transportRoutes`에서 조회하여 현실적인지 검증
   - 영업시간 제약 확인
3. 문제가 없으면 해당 `dayNItems` 배열에 시간순으로 아이템을 삽입한다
4. 식사 관련이면 `mealData` 배열도 업데이트한다
5. 예산이 바뀌면 BudgetSection 수치도 조정한다

### 3. 일정 수정 (Modify)

1. `page.tsx`를 읽어 수정 대상을 찾는다
2. Hard Constraints를 모두 검증한다
3. 시간 변경 시, 앞뒤 일정과의 간격을 확인한다
4. 해당 아이템을 수정하고 필요시 관련 데이터(mealData, budget 등)도 동기화한다

### 4. 일정 삭제 (Remove)

1. 삭제 대상 확인 후, 관련 데이터(mealData, mapUrl, transportCost 등) 정리
2. 빈 시간이 생기면 사용자에게 대안 활동을 제안한다

### 5. 시간 충돌 검사 (Conflict Check)

전체 일정 또는 특정 날의 시간 충돌을 분석:
- 아이템 간 이동시간 부족
- 영업시간 위반
- 전환 시간이 15분 미만인 구간
- Hard Constraints 위반

결과를 아래 형식으로 보고:
```
## 충돌 검사 결과

### Day N
- ⚠️ 12:00 점심 → 13:00 클라이밍: 이동시간 25분인데 간격 60분. OK
- ❌ 19:00 저녁 → 19:15 다음일정: 15분 미만 전환 (위험)
- ✅ 전체적으로 여유로움
```

## Editing page.tsx

`page.tsx` 수정 시 주의사항:
- `ScheduleItem` 타입에 맞는 필드만 사용할 것
- `time` 필드는 반드시 "HH:MM" 24시간 형식
- 배열 내 아이템은 시간 순으로 정렬 유지
- location 좌표는 Google Maps에서 정확한 값을 사용 (대략적 좌표도 OK)
- restaurant 필드는 식당일 때만 추가
- DaySchedule의 `mapUrl`은 Google Maps Directions URL로, 경유지가 바뀌면 함께 업데이트
- `transportCost`는 해당 날의 교통비 합계 문자열
